# 🏆 赛博俄罗斯方块：AI 与人类协作复盘 (Vibe Coding)

本项目是一个关于使用 Vite + React + TypeScript 快速构建具备现代音效与视觉反馈的游戏项目的复盘记录。通过 AI 的敏捷开发与人类的精准反馈，我们共同解决了一系列深层的 Web 开发与游戏逻辑难题。

## 🛠️ 技术挑战与解决方案仓库 (The Gotcha List)

### 1. 环境与基础调试 (The Invisible Barrier)
- **现象**: 初始部署后无法看到“开始游戏”入口。
- **根源**: 过度使用了 `overflow: hidden` 并结合了绝对居中。在不同分辨率下，CSS 虽然保证了居中，但由于没有滚动条且元素折叠，关键 UI 被“切”在了视口之外且无法触达。
- **经验**: 对于关键交互组件，应优先保证其在流式布局中的**展示优先级**，并显式保留溢出滚动 (`overflow-y: auto`)。

### 2. React 状态机的“幽灵”副作用 (Mutation Trap)
- **现象**: 方块在向左移动时会意外“瞬移”或者直接卡死在墙壁半空中。
- **根源**: 在 `useState` 的 updater 函数中使用了具有赋值副作用的操作：`pos: { x: (prev.pos.x += x) }`。
- **教训**: **永远不要在 React 函数式更新中进行原地修改 (In-place notation)**！在 React 的开发双重渲染 (Strict Mode) 下，该赋值会被执行两次，导致坐标翻倍漂移并穿透碰撞检测，引发后续逻辑坍塌。

### 3. 消行得分的“双倍惩罚” (Effect Loop)
- **现象**: 每消除 1 行，后台记录为 2 行，得分也翻倍。
- **根源**: 过于依赖 React 的响应式 Effect 链条。之前的设计是通过一个 `rowsCleared` 状态同步计分，并在 Effect 中监听。由于 React 开发模式的两次触发机制，该 Effect 在消行瞬间被跑了两遍。
- **教训**: 游戏的核心物理引擎逻辑（如：合并、消行、计分）具有高度的原子性，**应当采用确定性的同步函数回调**，而不是依赖异步监听状态变更。

### 4. 异步状态返回陷阱 (The Async Return trap)
- **现象**: 在 v2.2 版本重构后，消行突然失效，既无音效也无得分。
- **根源**: 在一个自定义 Hook 的 `useCallback` 中调用了 `setBoard(prev => { ... return sweptBoard })`。代码试图在执行完 `setBoard` 后立即 `return` 内部计算的行数。但 `setBoard` 的执行是异步的！函数在状态更新真正发生前就返回了默认的 `0`。
- **方案**: 将物理计算抽离为纯粹的工具函数 `getMergedAndSweptBoard`。在 `drop` 物理帧中同步获取计算结果，然后再统一更新 Board、Score 和播放音效。

## 💡 心得体会 (Dev Reflection)
- **UI 是一门“亮眼”的科学**: 为了实现“惊艳感”，我们引入了**极光绿 (Neon Green)** 的泛光样式。好的视觉反馈 (Visual Juice) 能极大抵消纯文字交互的枯燥感。
- **音效是灵魂**: 通过 `Web Audio API` 实时合成电子回馈，比加载 MP3 资源更符合游戏的瞬间响应直觉，成本也更低。
- **AI 敏捷之下的严谨**: AI 可以快速生成上百行组件，但其对 React 严格渲染逻辑的“微调”有时会出错。通过**精准的报错截图回传**与**逻辑分层（渲染层 vs 物理逻辑层）**，我们最终交付了具备生产环境健壮性的代码。

### 5. AI 自动化的“时机”陷阱
- **现象**: 开启 AI 模式后游戏不自动开始，或者方块中途开启无反应。
- **根源**: AI 按钮仅切换了状态位，但没有显式触发 `startGame` 函数；且算法限制了仅在 `y=0` 时计算。
- **方案**: 优化 `toggleAI` 函数，加入开局自检逻辑；移除决策的高度门槛，实现 AI 的“热接管”。
